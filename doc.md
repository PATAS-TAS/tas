# TAS - Telegram Anti-spam System

## Обзор системы

TAS - это автоматизированная система для обнаружения спама в группах Telegram. Система использует комбинацию методов, включая кэширование, быструю комплексную проверку, анализ мнений модераторов и проверку с помощью GPT для принятия решений о спам-сообщениях.

## Типы сообщений

### 1. checkMsg - сообщения для классификации
- Параметры: incoming: true, forwards: true
- Получение полных данных через API Telegram (включая медиа, тексты и цитаты)
- Предобработка: удаление первой текстовой строки каждого сообщения
- Объединение нескольких сообщений в один массив буфера для совместной обработки

### 2. sysMsg - системные сообщения
- Параметры: incoming: true, forwards: false
- Идентификация по наличию строку начинающейся с "Source:"
- Извлекаемые параметры:
  - "#r(\d+)" - reportID отчета
  - "😱(\d+)" - количество групп, пожаловавшихся на пользователя
  - Линии, начинающиеся с "Source:"
  - Линии, начинающиеся с "Sender:"

### 3. modMsg - системные сообщения модераторов
- Параметры: incoming: true, forwards: false
- Идентификация по наличию строки начинающейся с "Admin:"
- Извлекаемые параметры:
  - "#r(\d+)" - reportID отчета
  - " – Flood" или " – Not Spam" - мнение модератора

### 4. addMsg - дополнительные сообщения
- Параметры: incoming: true, forwards: false
- Типы сообщений и реакции:
  - "Hello there! Send /next to start processing reports." -> "/next 6"
  - "No Reports Found" -> выполнение функции undo()
  - "Please select 😡 BAN or 😌 NO." -> выполнение функции undo()
  - "Sorry, an error has occurred during your request. Please try again later." -> выполнение функции undo()
  - "Total this month:" - используется в проверке модераторов

## Главная логика работы

1. **Инициализация и настройка**:
   - Подключение к базам данных (PostgreSQL и Redis)
   - Инициализация Telegram клиента и бота
   - Настройка обработчиков событий для различных типов сообщений
   - Настройка периодических задач (сохранение данных, проверка здоровья системы)
   - Открытие приема отчетов

2. **Обработка входящих сообщений**:
   - `handleCheckMsg()`: Обрабатывает сообщения для проверки
   - `handleSysMsg()`: Обрабатывает системные сообщения
   - `handleAddMsg()`: Обрабатывает дополнительные сообщения
   - `handleModMsg()`: Обрабатывает системные сообщения модераторов

3. **Процесс обработки отчета** (`processReport()`):
   a) Быстрая комплексная проверка (`fastCheck()`)
   b) Если нет решения, проверка модераторов (`modCheck()`)
   c) Если нет решения, проверка с помощью GPT (`gptCheck()`)

4. **Открытие и закрытие отчетов**:
   - Открытие: В конце функции `main()` и в начале `applyDecision()`
   - Закрытие: При получении сообщения 'sysMsg'

5. **Сохранение результатов**:
   - Сохранение отчета и решения в кэше Redis
   - Периодическое сохранение данных из Redis в PostgreSQL

## Команды администратора

- "/start" - активация автоматического режима
- "/stop" - остановка работы приложения и переход в ручной режим
- "/status" - получение текущего статуса бота
- "/time [value]" - установка задержки между командами (в миллисекундах)
- "/reset" - очистка кэша Redis
- "/db" - выполнение операций с базой данных и генерация отчета

## Процесс определения спама

1. **Быстрая комплексная проверка**:
   - Проверка кэша на наличие идентичного сообщения или медиа
   - Проверка на наличие ссылок, @юзернеймов, контактов и медиа с более чем 2 жалобами
   - Проверка на наличие инлайн-кнопок, историй, пересланных историй или цитат

2. **Проверка модераторов**:
   - Отправка "/stats" и ожидание addMsg с помощью функции `waitStats()`
   - Отправка reportId и ожидание adminMsg с помощью функции `waitUpdated()`
   - Анализ мнений модераторов и сохранение решения в кеше Redis 
   - Отправка "/next 2"
   - Отправка результата решения с помощью проверки fastCheck (checkCache)

3. **Проверка с помощью GPT**:
   - Отправка сообщения и контекста (количество жалоб, источник, отправитель) в GPT
   - Анализ ответа GPT и принятие решения

## Система анализа ответа модераторов

- 2 или более "– Flood" -> Спам (100% уверенность)
- 2 или более "– Not Spam" -> Не спам (100% уверенность)
- 1 "– Flood" и 0 "– Not Spam" -> Спам (90% уверенность)
- 1 "– Not Spam" и 0 "– Flood" -> Не спам (90% уверенность)
- 1 "– Flood" и 1 "– Not Spam", нет мнений или другие неоднозначные комбинации -> Нет решения и передается другим проверкам

## Обработка ошибок и восстановление

- При возникновении ошибок выполняется функция `undo()`
- Система хранит идентификаторы последних 10 обработанных отчетов для возможности отмены
- Реализован механизм повторных попыток при ошибках сети или недоступности Telegram API

## Кэширование и работа с базой данных

- Использование Redis для быстрого кэширования отчетов и решений
- Использование LRU Cache для хранения мнений модераторов (TTL: 1 минута)
- Периодическое сохранение данных из Redis в PostgreSQL (каждые 2 часа)
- Сохранение отчетов в PostgreSQL с партиционированием по дате

## Особенности реализации GPT проверки

### Текстовая проверка (GPT-4)
- Использует улучшенный промпт для анализа коммерческого спама
- Учитывает контекст, метаданные и содержимое сообщения
- Возвращает классификацию (спам/не спам) и уровень уверенности

### Проверка медиа (GPT-4 Vision)
- Анализирует изображения на предмет признаков коммерческого спама
- Использует специальный промпт для оценки визуального контента
- Возвращает только классификацию (спам/не спам)

### Принятие решения
- Приоритет отдается результатам анализа медиа, если обнаружен спам
- При отсутствии спама в медиа используется результат текстового анализа
- Если текстовый анализ не проводился, используется результат анализа медиа

- Автоматическое удаление устаревших медиа-файлов из Redis
