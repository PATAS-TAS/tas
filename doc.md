# TASmod - логика модераторской системы

## Общий процесс работы

1. Инициализация Telegram клиента, подключение к PostgreSQL и Redis.
2. Отправка администратору информации о готовности системы.
3. Администратор отправляет команду "/start" для активации автоматического режима.
4. Система активирует автоматический режим и отправляет на botId команду "/next".
5. Получение и обработка отчетов от бота (botId).

## Типы сообщений

### 1. checkMsg - сообщения для классификации
- Параметры: incoming: true, forwards: true
- Получение полных данных через API Telegram (включая медиа, тексты и цитаты)
- Предобработка: удаление первой текстовой строки каждого сообщения
- Объединение нескольких сообщений в один массив буфера для совместной обработки

### 2. sysMsg - системные сообщения
- Параметры: incoming: true, forwards: false
- Идентификация по эмодзи с цифрами "/😱(\d+)/"
- Извлекаемые параметры:
  - "r#(\d+)" - reportID отчета
  - "/😱(\d+)/" - количество групп, пожаловавшихся на пользователя
  - Линии, начинающиеся с "🗣 Source: " или "Source: " (нам не нужно извлекать само слово "Source: ", нам нужно то, что идет после него)
  - Линии, начинающиеся с "Sender: "
  - Вероятность спама по мнению Telegram: "/(?:🌕|🌔|🌓|🌒|🌚)\s*(\d+)%/"
  - "🔴 " - наличие внешней ссылки
  - "🔶 " - наличие внутренней ссылки на Telegram канал или группу
  - " – Flood" или " – Not Spam" - мнение модератора

### 3. addMsg - дополнительные сообщения
- Параметры: incoming: true, forwards: false
- Типы сообщений и реакции:
  - "Hello there! Send /next to start processing reports." -> "/next"
  - "No Reports Found" -> "/undo"
  - "Please select 😡 BAN or 😌 NO." -> "/undo"
  - "Sorry, an error has occurred during your request. Please try again later." -> "/undo"
  - "Your Fee for this month:" - информация о заработке

## Команды администратора

- "/start" - активация автоматического режима
- "/stop" - остановка работы приложения и переход в ручной режим
- "/time [миллисекунды]" - установка задержки обработки сообщений (по умолчанию 100 мс)
- "/status" - получение текущего статуса бота

## Процесс обработки отчетов

1. Проверка кэша Redis на наличие идентичных сообщений.
2. Если найдено в кэше, отправка сохраненного решения без проверки модераторов. (Запись в кэш об этом делать НУЖНО ВСЕ РАВНО)
3. Если не найдено в кэше:
   a) Получение сообщения для проверки и его reportID.
   b) Отправка "/stats" и получение информации о заработке.
   c) Отправка reportID и получение системного сообщения.
   d) Извлечение данных из системного сообщения.
   e) Принятие решения на основе мнений модераторов.
   f) Отправка "/next".
   g) Отправка результата проверки: "😡 SPAM" или "😌 NO".
4. Сохранение результата в Redis для кэширования.
5. Если есть два одинаковых мнения модераторов, сохранение данных в Redis для последующей выгрузки в PostgreSQL. (нужно отмечать как-то, какие данные мы будем выгружать в 23:00 каждый день. Но по команде /db от админа, кэш Redis в тот же момент выгружается в Postgre, затем создаётся csv файл и отправляется админу)

## Правила принятия решений

- 1 "– Flood" -> "😡 SPAM"
- 2 "– Flood" -> "😡 SPAM" (сохранение в Redis)
- 1 "– Not Spam" -> "😌 NO"
- 2 "– Not Spam" -> "😌 NO" (сохранение в Redis)
- 1 "– Flood" и 1 "– Not Spam" -> "😌 NO"
- Нет мнений -> "/undo"

## Дополнительные функции

1. Асинхронное сохранение в PostgreSQL: ежедневно в 23:00 все данные из Redis выгружаются в PostgreSQL.
2. Мониторинг производительности: если кэш не смог выгрузиться в бд, нужно уведомить администратора.
3. Система уведомлений администратора: отправка важных уведомлений через Telegram.
4. Расширенная обработка ошибок: логирование и уведомление администратора о критических проблемах.
5. Graceful shutdown: корректное завершение работы приложения при получении сигналов SIGTERM и SIGINT.

## Оптимизации

1. Удаление первой строки каждого сообщения перед добавлением в буфер.
2. Сокращенные названия функций для улучшения читаемости кода.
3. Оптимизированная обработка сообщений и системной информации.
4. Улучшенное кэширование результатов проверок.
5. Эффективное использование буфера для обработки нескольких сообщений.

## Важные замечания

1. При отсутствии мнений модераторов или наличии только одного мнения решение принимается, но не сохраняется в базу данных.
2. В кэш в одну строку сохраняются проверяемое сообщение, метаданные из системного сообщения и наша классифкация. Эти же данные выгружаются в бд.
3. В бд загружаются только те классификации, у которых было два одинаковых мнения модераторов.

## Описание необходимости основных функций

1. `initClient()`: Необходима для установления соединения с Telegram API. Это основа всей системы, без которой невозможна работа бота.

2. `sendToBot()`: Критически важна для взаимодействия с ботом модерации. Позволяет отправлять команды и получать отчеты.

3. `notifyAdmin()`: Обеспечивает оперативное информирование администратора о важных событиях и ошибках, что критично для мониторинга и быстрого реагирования.

4. `handleAdminMsg()`: Позволяет администратору управлять ботом удаленно, что необходимо для гибкости в работе системы.

5. `checkCache()`: Ускоряет работу системы, предотвращая повторную обработку уже проверенных сообщений.

6. `processReport()`: Центральная функция системы, отвечающая за обработку отчетов о спаме. Без нее невозможно принятие решений.

7. `makeDecision()`: Реализует логику принятия решений на основе мнений модераторов. Ключевая для автоматизации процесса модерации.

8. `handleCheckMsg()`: Обрабатывает входящие сообщения для проверки, что необходимо для инициации процесса модерации.

9. `handleSysMsg()`: Извлекает важную системную информацию из сообщений бота, что критично для анализа и принятия решений.

10. `handleAddMsg()`: Обрабатывает дополнительные сообщения от бота, обеспечивая корректную работу системы в различных ситуациях.

11. `getStatus()`: Предоставляет текущий статус бота, что необходимо для мониторинга и отладки.

12. `healthCheck()`: Обеспечивает стабильность работы системы, своевременно выявляя и устраняя проблемы.

13. `saveToPostgres()`: Сохраняет данные в долгосрочное хранилище, что необходимо для анализа и отчетности.

14. `main()`: Основная функция, инициализирующая и координирующая работу всей системы.

15. `gracefulShutdown()`: Обеспечивает корректное завершение работы системы, предотвращая потерю данных и зависание процессов.

Каждая из этих функций играет важную роль в обеспечении эффективной, надежной и автоматизированной работы системы модерации. Их взаимодействие позволяет достичь высокой точности в обнаружении спама при сохранении быстродействия и масштабируемости системы.
