# TAS - Telegram Anti-spam System

## Обзор системы

TAS - это автоматизированная система для обнаружения и управления спамом в группах Telegram. Система использует комбинацию методов, включая кэширование, обнаружение очевидного спама, анализ на основе GPT и мнения модераторов для принятия решений о спам-сообщениях.

## Типы сообщений

### 1. checkMsg - сообщения для классификации
- Параметры: incoming: true, forwards: true
- Получение полных данных через API Telegram (включая медиа, тексты и цитаты)
- Предобработка: удаление первой текстовой строки каждого сообщения
- Объединение нескольких сообщений в один массив буфера для совместной обработки

### 2. sysMsg - системные сообщения
- Параметры: incoming: true, forwards: false
- Идентификация по эмодзи с цифрами "/😱(\d+)/"
- Извлекаемые параметры:
  - "r#(\d+)" - reportID отчета
  - "/😱(\d+)/" - количество групп, пожаловавшихся на пользователя
  - Линии, начинающиеся с "🗣 Source: " или "Source: "
  - Линии, начинающиеся с "Sender: "
  - Вероятность спама по мнению Telegram: "/(?:🌕|🌔|🌓|🌒|🌚)\s*(\d+)%/"
  - "🔴 " - наличие внешней ссылки
  - "🔶 " - наличие внутренней ссылки на Telegram канал или группу
  - " – Flood" или " – Not Spam" - мнение модератора

### 3. addMsg - дополнительные сообщения
- Параметры: incoming: true, forwards: false
- Типы сообщений и реакции:
  - "Hello there! Send /next to start processing reports." -> "/next"
  - "No Reports Found" -> "/undo"
  - "Please select 😡 BAN or 😌 NO." -> "/undo"
  - "Sorry, an error has occurred during your request. Please try again later." -> "/undo"
  - "Your Fee for this month:" - используем в проверке модераторов

## Главная логика работы

1. **Инициализация и настройка**:
   - Подключение к базам данных (PostgreSQL и Redis)
   - Инициализация Telegram клиента и бота
   - Настройка обработчиков событий для различных типов сообщений

2. **Обработка входящих сообщений**:
   - `handleCheck()`: Обрабатывает сообщения для проверки
     - Удаляет первую строку каждого сообщения (используя `preprocessMessage`)
     - Собирает содержимое сообщения и хеши медиафайлов
   - `handleSys()`: Обрабатывает системные сообщения
     - Использует `parseSysMsg()` для извлечения информации
     - Обновляет текущий отчет или создает новый
   - `handleAddMsg()`: Обрабатывает дополнительные сообщения
     - Реагирует на специфические сообщения от бота

3. **Процесс обработки отчета** (`processReport()`):
   a) Проверка кэша:
      - Если отчет найден в кэше, используется сохраненное решение
   b) Проверка на очевидный спам:
      - Применяются автоматические критерии определения спама
   c) Проверка мнений модераторов:
      - Запрашивает и анализирует мнения модераторов
      - Принимает решение на основе правил для мнений модераторов
   d) GPT анализ:
      - Если включен, отправляет содержимое сообщения на анализ GPT
   e) Принятие окончательного решения:
      - Если ни один метод не дал уверенного результата, отправляется "/undo"
   f) Сохранение результата:
      - Сохраняет решение в Redis и отправляет боту

4. **Параллельная обработка**:
   - Проверка на очевидный спам и проверка кэша выполняются параллельно
   - Используется асинхронное программирование для эффективной обработки

5. **Управление данными**:
   - Периодическая синхронизация данных между Redis и PostgreSQL
   - Экспорт данных в CSV по запросу администратора

6. **Мониторинг и управление**:
   - Периодические проверки состояния баз данных и соединения с Telegram
   - Обработка команд администратора для управления системой

## Команды администратора

- "/start" - активация автоматического режима
- "/stop" - остановка работы приложения и переход в ручной режим
- "/time [миллисекунды]" - установка задержки обработки сообщений (по умолчанию 1000 мс)
- "/delay [миллисекунды]" - установка задержки между командами (по умолчанию 150 мс)
- "/status" - получение текущего статуса бота
- "/db" - запрос экспорта базы данных в CSV
- "/toggle [obvs/cache/gpt/mods]" - включение/выключение отдельных этапов проверки
- "/correct #rXXXXX" - исправление решения для конкретного отчета
- "/reset" - очистка кэша Redis

## Процесс исправления решений

- Администратор отправляет команду "/correct #rXXXXXX", где XXXXXX - это номер отчета.
- Система находит соответствующий отчет в кэше Redis.
- Если отчет найден, система изменяет решение на противоположное (спам на не спам или наоборот).
- Обновленный отчет сохраняется в кэше Redis и в базе данных PostgreSQL.
- Поле corrected устанавливается в true для отмеченного отчета.
- Администратор получает уведомление о выполненном исправлении.

Эта функция позволяет исправлять ошибочные решения и может быть использована для улучшения точности системы в будущем.

## Автоматические критерии определения спама

1. Чрезмерное использование эмодзи или необычное форматирование текста (более 30% от общего содержания сообщения).

2. Повторяющиеся элементы в сообщении:
   - Повторяющиеся слова или фразы (более 3 повторений).
   - Длинные последовательности повторяющихся символов (5 или более).

3. Множественные ссылки и контактная информация:
   - Более 2 ссылок в одном сообщении (включая ссылки на Telegram-каналы).
   - Одновременное наличие нескольких @username и ссылки на канал Telegram.

4. Чрезмерное использование заглавных букв (более 50% текста при длине сообщения более 20 символов).

5. Подозрительная структура сообщения:
   - Множественные строки с одним словом (более 5 таких строк).
   - Длинные сообщения (более 500 символов) со ссылками.

6. Наличие медиа, ссылок или контактной информации в сообщении с более чем 2 жалобами.

7. Наличие содержимого типа "story" в сообщении.

8. Наличие потенциально опасных типов файлов (например, .exe, .apk, .bat).

Эти критерии направлены на выявление структурных и форматных особенностей спам-сообщений, минимизируя риск ложных срабатываний, связанных с конкретным содержанием сообщений.

## Правила принятия решений (после обработки мнений модераторов)

- 2 или более "– Flood" -> "😡 SPAM" (reason: "MODERATOR_2_FLOOD")
- 2 или более "– Not Spam" -> "😌 NO" (reason: "MODERATOR_2_NOT_SPAM")
- 1 "– Flood" и 0 "– Not Spam" -> "😡 SPAM" (reason: "MODERATOR_1_FLOOD")
- 1 "– Not Spam" и 0 "– Flood" -> "😌 NO" (reason: "MODERATOR_1_NOT_SPAM")
- 1 "– Flood" и 1 "– Not Spam" -> GPT
- Нет мнений -> Используется оценка GPT
- Любая другая комбинация -> Продолжение проверки с использованием GPT

## Интеграция с GPT

- Выбор модели GPT в зависимости от длины сообщения.
- Анализ сообщения с учетом контекста (количество жалоб, название группы, отправитель).
- GPT возвращает два числа: классификацию (0 для не-спама или 1 для спама) и уровень уверенности (от 0 до 100).
- При отсутствии сообщения для проверки, GPT анализирует доступную системную информацию.
- При получении неожиданного ответа (не "0" и не "1") или в случае ошибки:
  - Отправляется команда "/undo"
  - Администратор уведомляется о проблеме
  - Этап проверки GPT пропускается
- Принятие решения на основе ответа GPT при отсутствии явных признаков спама и мнений модераторов.
- Сохранение результата классификации и уровня уверенности в отчете.

## Система автоматического обновления списка ключевых слов для спама

1. Ежедневное обновление списка топ-100 спам-фраз на основе данных из PostgreSQL.
2. Анализ спам-сообщений и не-спам сообщений за последние 30 дней.
3. Использование метода N-грамм (триграмм) для выделения фраз.
4. Подсчет частоты встречаемости фраз в спам и не-спам сообщениях.
5. Вычисление взвешенной оценки для каждой фразы.
6. Хранение актуального списка спам-фраз в Redis для быстрого доступа.
7. Использование обновленного списка при проверке новых сообщений на спам.

## Процесс определения топ-100 спам фраз

1. Выборка до 10000 спам-сообщений и 10000 не-спам сообщений из базы данных за последние 30 дней.
2. Токенизация каждого сообщения и приведение к нижнему регистру.
3. Создание триграмм (последовательностей из трех слов) из токенизированного текста.
4. Подсчет частоты встречаемости каждой триграммы в спам и не-спам сообщениях.
5. Вычисление взвешенной оценки для каждой фразы с учетом её специфичности для спама.
6. Сортировка фраз по взвешенной оценке и выбор топ-100.

## Дополнительные функции

1. Асинхронное сохранение в PostgreSQL: ежедневно в 23:00 все данные из Redis выгружаются в PostgreSQL.

## Важные замечания

1. При отсутствии мнений модераторов решение принимается на основе оценки GPT.
2. В кэш сохраняются проверяемое сообщение, метаданные из системного сообщения и результат классификации (isSpam). 
3. Данные в кэше Redis и в PostgreSQL синхронизируются при каждом сохранении отчета.
4. Вероятность спама от Telegram используется как второстепенный фактор в анализе.
5. Название группы (Source) используется для понимания контекста, а не как индикатор спама.

## Процесс возврата к предыдущим отчетам

Система реализует механизм возврата к предыдущим отчетам для обеспечения непрерывности обработки в случае возникновения проблем или отсутствия необходимой информации.

1. Система хранит идентификаторы последних 10 обработанных отчетов в Redis.
2. При необходимости возврата, система перебирает сохраненные идентификаторы отчетов.
3. Для каждого идентификатора отправляется команда "/undo" с добавлением числового идентификатора отчета (например, "/undo233073558").
4. Система пытается вернуться к предыдущему отчету, пока не получит валидный отчет для обработки или не исчерпает все сохраненные идентификаторы.
5. Если не удается вернуться ни к одному из последних 10 отчетов:
   - Администратор уведомляется о проблеме
   - Система продолжает работу, переходя к следующему доступному отчету

## Проверка мнений модераторов

При проверке мнений модераторов система теперь ожидает получения сообщения "Your Fee for this month:" перед отправкой идентификатора отчета. Это обеспечивает правильную последовательность обработки и получения всей необходимой информации от бота.

## Оптимизация базы данных

### Партиционирование таблицы reports

Для оптимизации производительности и управления данными, таблица `reports` партиционирована по дате создания записи (поле `created_at`). Это позволяет:

1. Улучшить производительность запросов, особенно для больших объемов данных.
2. Упростить архивирование и удаление старых данных.
3. Более эффективно управлять индексами.

Структура партиционированной таблицы:

```sql
CREATE TABLE reports (
  id SERIAL,
  report_id TEXT,
  message_content TEXT[],
  media_hashes TEXT[],
  complaint_count INTEGER NOT NULL,
  source TEXT NOT NULL,
  sender TEXT NOT NULL,
  is_spam INTEGER,
  reason TEXT,
  corrected BOOLEAN DEFAULT FALSE,
  confidence FLOAT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);
```

Ежемесячные партиции создаются автоматически при инициализации базы данных для текущего года.

## Адаптивное архивирование данных

Система включает в себя функцию адаптивного архивирования `adaptiveArchiving`, которая автоматически архивирует данные на основе текущего использования базы данных:

1. Система ежедневно проверяет текущий размер базы данных.
2. Если использование базы данных превышает заданный порог (по умолчанию 80% от максимального размера), запускается процесс архивирования.
3. Архивируются самые старые данные, необходимые для снижения использования базы данных ниже порогового значения.
4. Архивные данные экспортируются в CSV-файл и отправляются администратору через Telegram.
5. После успешной отправки файла администратору, архивные данные удаляются из основной таблицы.
6. Локальный CSV-файл удаляется после отправки.

### Процесс адаптивного архивирования:

1. Проверка текущего размера базы данных с помощью функции `checkDatabaseSize`.
2. Если размер превышает пороговое значение, запускается функция `archiveOldData`.
3. Определение даты, с которой нужно начать архивирование для оптимизации размера базы данных.
4. Выборка данных для архивирования из базы данных.
5. Создание CSV-файла с архивными данными.
6. Отправка CSV-файла администратору через Telegram.
7. Удаление архивированных данных из основной таблицы базы данных.
8. Удаление локального CSV-файла.

- `DB_ARCHIVE_THRESHOLD`: Пороговое значение использования базы данных, при котором начинается архивирование (по умолчанию 0.8 или 80%).

### Расписание:

Проверка размера базы данных и адаптивное архивирование выполняются ежедневно в 2:00 AM.

### Обработка ошибок:

- Все ошибки в процессе проверки размера базы данных и архивирования логируются для дальнейшего анализа.
- В случае ошибки при отправке файла администратору, система отправляет уведомление об ошибке.

## Обработка ошибок валидации и невалидных отчетов

Система включает в себя механизм обработки ошибок валидации и хранения невалидных отчетов для дальнейшего анализа и устранения проблем.

### Процесс валидации и обработки ошибок

1. Каждый отчет проходит валидацию перед сохранением в базу данных.
2. Если отчет не проходит валидацию:
   - Ошибка валидации логируется с детальной информацией о невалидном отчете.
   - Администратор уведомляется о проблеме валидации через Telegram.
   - Невалидный отчет сохраняется в отдельную структуру в Redis для дальнейшего анализа.

### Хранение невалидных отчетов

- Невалидные отчеты сохраняются в Redis с ключом вида `invalid_report:{report_id}`.
- Каждая запись содержит:
  - Полные данные невалидного отчета
  - Сообщение об ошибке валидации
  - Временную метку сохранения
- Невалидные отчеты хранятся в течение 7 дней для возможности их анализа.

### Периодическая проверка невалидных отчетов

- Система выполняет автоматическую проверку наличия невалидных отчетов каждые 6 часов.
- Если обнаружены невалидные отчеты, администратор получает уведомление с количеством таких отчетов.

### Анализ и устранение проблем

- Администраторы могут анализировать невалидные отчеты для выявления систематических проблем в сборе или обработке данных.
- На основе анализа невалидных отчетов могут быть внесены изменения в систему для предотвращения подобных ошибок в будущем.

## Обработка ошибок и восстановление

1. При возникновении критических ошибок или невозможности продолжить обработку отчетов, система:
   - Уведомляет администратора о возникшей проблеме
   - Приостанавливает работу на 2 минуты для возможного вмешательства администратора
   - Автоматически возобновляет работу после паузы, если не было ручного вмешательства

2. Система использует Redis для эффективного хранения и быстрого доступа к идентификаторам последних обработанных отчетов, что позволяет оперативно восстанавливать работу в случае сбоев.

3. Механизм паузы и автоматического возобновления работы обеспечивает баланс между попытками самовосстановления системы и возможностью ручного вмешательства администратора в случае серьезных проблем.

4. Система имеет механизм повторных попыток подключения к базе данных:
   - При инициализации базы данных выполняется до 5 попыток подключения с интервалом в 5 секунд
   - Если все попытки неудачны, система уведомляет администратора и завершает работу
   - Периодически проверяется состояние соединения с базой данных, и при необходимости выполняется попытка восстановления соединения

5. Добавлена новая функция `checkAndRestoreDBConnection`, которая:
   - Регулярно проверяет соединение с базой данных
   - При потере соединения пытается восстановить его
   - Уведомляет администратора о проблемах с соединением

Эти механизмы обеспечивают высокую отказоустойчивость системы и минимизируют время простоя в случае возникновения проблем с базой данных.
