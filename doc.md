# TAS - Telegram Anti-spam System

## Типы сообщений

### 1. checkMsg - сообщения для классификации
- Параметры: incoming: true, forwards: true
- Получение полных данных через API Telegram (включая медиа, тексты и цитаты)
- Предобработка: удаление первой текстовой строки каждого сообщения
- Объединение нескольких сообщений в один массив буфера для совместной обработки

### 2. sysMsg - системные сообщения
- Параметры: incoming: true, forwards: false
- Идентификация по эмодзи с цифрами "/😱(\d+)/"
- Извлекаемые параметры:
  - "r#(\d+)" - reportID отчета
  - "/😱(\d+)/" - количество групп, пожаловавшихся на пользователя
  - Линии, начинающиеся с "🗣 Source: " или "Source: "
  - Линии, начинающиеся с "Sender: "
  - Вероятность спама по мнению Telegram: "/(?:🌕|🌔|🌓|🌒|🌚)\s*(\d+)%/"
  - "🔴 " - наличие внешней ссылки
  - "🔶 " - наличие внутренней ссылки на Telegram канал или группу
  - " – Flood" или " – Not Spam" - мнение модератора

### 3. addMsg - дополнительные сообщения
- Параметры: incoming: true, forwards: false
- Типы сообщений и реакции:
  - "Hello there! Send /next to start processing reports." -> "/next"
  - "No Reports Found" -> "/undo"
  - "Please select 😡 BAN or 😌 NO." -> "/undo"
  - "Sorry, an error has occurred during your request. Please try again later." -> "/undo"
  - "Your Fee for this month:" - используем в проеверке модераторов

## Команды администратора

- "/start" - активация автоматического режима
- "/stop" - остановка работы приложения и переход в ручной режим
- "/time [миллисекунды]" - установка задержки обработки сообщений (по умолчанию 1000 мс)
- "/status" - получение текущего статуса бота
- "/db" - запрос экспорта базы данных в CSV

## Процесс обработки отчетов

1. Параллельное выполнение:
   a) Автоматическая проверка на очевидный спам по заданным критериям.
   b) Проверка кэша Redis на наличие идентичных сообщений.
2. Если сообщение определено как очевидный спам, оно сразу классифицируется и сохраняется.
3. Если сообщение найдено в кэше и идентично (по содержанию сообщения, медиа и количеству жалоб), используется сохраненное решение.
4. Если не определено как спам и не найдено в кэше:
   a) Отправка "/stats" и получение информации о заработке.
   b) Отправка reportID и получение системного сообщения.
   c) Обработка мнений модераторов:
      - Ожидание мнений модераторов (максимум 30 секунд, таймер сбрасывается).
      - Анализ полученных мнений и принятие решения.
   d) Отправка "/next".
   e) Отправка результата проверки: "😡 SPAM" или "😌 NO".
5. Сохранение результата в Redis (с дальнейшей выгрузкой в PostgreSQL)

## Автоматические критерии определения спама

1. Наличие медиа, ссылок, номеров телефонов или @юзернеймов в сообщении с более чем 2 жалобами.
2. Все истории (stories) считаются спамом по умолчанию.
3. Наличие URL кнопок в сообщении.
4. Дублирование сообщений (более 70% идентичных сообщений в отчете).
5. Повторяющиеся ссылки или юзернеймы в одном сообщении.
6. Наличие файлов с потенциально опасными расширениями.

## Правила принятия решений (после обработки мнений модераторов)

- 2 или более "– Flood" -> "😡 SPAM"
- 2 или более "– Not Spam" -> "😌 NO"
- 1 "– Flood" и 0 "– Not Spam" -> "😡 SPAM"
- 1 "– Not Spam" и 0 "– Flood" -> "😌 NO"
- 1 "– Flood" и 1 "– Not Spam" -> "😌 NO"
- Нет мнений -> "/undo"
- Любая другая комбинация -> "😌 NO" (с логированием неожиданной ситуации)

## Дополнительные функции

1. Асинхронное сохранение в PostgreSQL: ежедневно в 23:00 все данные из Redis выгружаются в PostgreSQL.
2. Мониторинг производительности: если кэш не смог выгрузиться в бд, уведомляется администратор.
3. Система уведомлений администратора: отправка важных уведомлений через Telegram.
4. Расширенная обработка ошибок: логирование и уведомление администратора о критических проблемах.
5. Graceful shutdown: корректное завершение работы приложения при получении сигналов SIGTERM и SIGINT.

## Оптимизации

1. Удаление первой строки каждого сообщения перед добавлением в буфер.
2. Сокращенные названия функций для улучшения читаемости кода.
3. Оптимизированная обработка сообщений и системной информации.
4. Улучшенное кэширование результатов проверок с использованием Redis.
5. Эффективное использование буфера для обработки нескольких сообщений.
6. Автоматическая проверка на спам перед проверкой кэша и обращением к модераторам.
7. Параллельное выполнение проверки на очевидный спам и проверки кэша для ускорения обработки отчетов.

## Важные замечания

1. При отсутствии мнений модераторов решение не принимается, отправляется "/undo".
2. В кэш сохраняются проверяемое сообщение, метаданные из системного сообщения и результат классификации (isSpam). 
3. Данные в кэше Redis и в PostgreSQL синхронизируются при каждом сохранении отчета.

## Технические детали

- Язык программирования: TypeScript
- Основные библиотеки: telegram, ioredis, pg, express, node-schedule, winston
- База данных: PostgreSQL для долгосрочного хранения, Redis для кэширования
- API: Telegram Bot API, собственный REST API для мониторинга

- Использование переменных окружения для хранения чувствительных данных

- Проверка на потенциально опасные файлы

- Возможность горизонтального масштабирования за счет использования Redis для кэширования
- Оптимизация запросов к базам данных для повышения производительности при росте объема данных
- Автоматическое определение спама для уменьшения нагрузки на модераторов

## Описание основных функций

1. `initClient()`: Инициализация клиента Telegram.
2. `initDB()`: Инициализация базы данных PostgreSQL.
3. `initBot()`: Инициализация бота Telegram.
4. `handleCheck()`: Обработка проверочных сообщений.
5. `handleSys()`: Обработка системных сообщений.
6. `parseSysMsg()`: Парсинг системных сообщений.
7. `validateReport()`: Валидация отчета.
8. `checkCache()`: Проверка кэша Redis на наличие идентичных сообщений.
9. `saveToCache()`: Сохранение в кэш Redis.
10. `processReport()`: Обработка отчета.
11. `makeDecision()`: Принятие решения о спаме на основе мнений модераторов.
12. `sendDecision()`: Отправка решения боту.
13. `saveReport()`: Сохранение отчета в базу данных и кэш.
14. `notify()`: Уведомление администратора.
15. `handleAdmin()`: Обработка команд администратора.
16. `startAutoMode()`: Запуск автоматического режима.
17. `stopAutoMode()`: Остановка автоматического режима.
18. `sendStatus()`: Отправка статуса системы.
19. `handleDbExport()`: Обработка экспорта базы данных.
20. `setupHandlers()`: Настройка обработчиков событий.
21. `exportCSV()`: Экспорт данных в CSV.
22. `checkDB()`: Проверка соединения с базой данных.
23. `checkDBSettings()`: Проверка настроек базы данных.
24. `checkDBContent()`: Проверка содержимого базы данных.
25. `sendToBot()`: Отправка сообщения боту.
26. `getReportId()`: Получение reportId от бота.
27. `transferDataFromRedisToPostgres()`: Передача данных из Redis в PostgreSQL.
28. `main()`: Основная функция приложения.
29. `detectObviousSpam()`: Автоматическая проверка на спам.
30. `processModeratorsOpinions()`: Обработка мнений модераторов.
31. `ensureMinimumInterval()`: Обеспечение минимального интервала между обработкой отчетов.
32. `gracefulShutdown()`: Корректное завершение работы приложения.

## Процесс запуска

1. Запуск функции `main()`.
2. Инициализация базы данных, клиента Telegram и бота.
3. Настройка обработчиков событий.
4. Запуск сервера Express.
5. Настройка периодических задач (проверка базы данных, передача данных из Redis в PostgreSQL).
6. Ожидание команд администратора для начала работы.

## Обработка ошибок

- Уведомление администратора о критических ошибках.
- Graceful shutdown при получении сигналов SIGINT и SIGTERM.

## Мониторинг

- Периодическая проверка соединения с базой данных.
- Отправка статуса системы администратору по запросу.
- Логирование важных событий и ошибок.

## Требования к окружению

- Node.js версии 18 или выше
- PostgreSQL
- Redis
- Доступ к API Telegram

## Переменные окружения

- `DEEP_LOG`: Включение детального логирования (true/false)
- `PORT`: Порт для сервера Express
- `ADMIN_ID`: ID администратора в Telegram
- `DATABASE_URL`: URL подключения к PostgreSQL
- `REDIS_URL`: URL подключения к Redis
- `API_ID`: API ID для Telegram
- `API_HASH`: API Hash для Telegram
- `SESSION_STRING`: Строка сессии для Telegram клиента
- `BOT_ID`: ID бота в Telegram
- `BOT_ACCESS_HASH`: Access Hash бота в Telegram
- `DB_CHECK_INTERVAL`: Интервал проверки базы данных (в миллисекундах)
- `HEALTH_CHECK_INTERVAL`: Интервал проверки здоровья системы (в миллисекундах)
- `PROCESSING_INTERVAL`: Интервал обработки отчетов (в миллисекундах)
